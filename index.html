The Fundamentals of Healthcare Interoperability
for Measure Developers

In the US, most healthcare providers have adopted an electronic healthcare record (EHR) or other software for capturing clinical information. This isn't a silver bullet though. Healthcare data can remain trapped within these systems. Even today, clinicians still fall back on old and inefficient ways of sharing clinical information from faxing chart notes to manually sharing paper-based documents. Without the seamless exchange of healthcare data, it can be difficult for clinicians caring for a patient to get the complete picture of the personâ€™s current health. This lack of coordination in the care of patients impacts the ability of clinicians to make appropriate clinical decisions and negatively impacts patient care. There are numerous examples where this lack of access to data has led to actual patient harm including death.

On this page, we will try to demystify some of the terms and acronyms that measure developers may come across in this world.

Standards

A standard is simply a commonly agreed way of doing something. And, a data standard is an agreement related to the representation, format, definition, structuring, tagging, transmission, manipulation, use, and management of data. In our modern world, the adoption of data standards has allowed for surfing the world-wide web, making purchases online, and even reserving a flight using a third-party website. While these activities are still possible without standards, it is considerably more difficult for a vendor to deal with numerous different interfaces rather than just one.

Health Level Seven International (HL7)

HL7 is a non-profit organization dedicated to developing standards for the exchange of electronic healthcare data. The organization is supported by more than 1,600 members from over 50 countries, including 500+ corporate members representing healthcare providers, government stakeholders, payers, pharmaceutical companies, vendors/suppliers, and consulting firms.

Over the years, HL7 has produced several standards to facilitate the exchange of healthcare information, including Clinical Document Architecture (CDA), and the Fast Healthcare Interoperability Resources (FHIR) standard.

Clinical Document Architecture (CDA)

CDA is an interoperability standard that was released as part of HL7 V3 but had some challenges that limited implementation:

  • Steep learning curve
  • Complexity limited true interoperability
  • CDA documents do not fit well in many workflows
  • Extensibility is difficult

Fast Healthcare Interoperability Resources (FHIR)

FHIR (pronounced "fire") is an interoperability standard intended to facilitate the exchange of healthcare information between healthcare providers, patients, caregivers, payers, researchers, and anyone else involved in the healthcare ecosystem.

The standard focuses on speed to learn and implement, lowering costs, scaling well from simple to complex, flexibility, and keeping the standard free to use. And, it leverages existing web technologies such as HTTP, REST, JSON, XML, and OAuth. These technologies should be familiar to most developers, reducing the learning curve compared with other standards.

Technology Description
Hyper Text Transfer Protocol (HTTP) The protocol allows for communication between clients (e.g., browser) and servers on the World Wide Web. The primary or most commonly-used HTTP methods are POST, GET, PUT, PATCH, and DELETE. These methods correspond to create, read, update, and delete (or CRUD) operations, respectively.
REpresentational State Transfer (REST)

The RESTful API architecture is built on the following guiding principles:

  • Uniform interface for identifying and manipulating resources
  • Separation of concerns between the client and server so each can evolve independently
  • Stateless in that it requires that requests from the client to the server contain all information necessary to complete the request

Note: An application user programming interface (API) defines how two software applications communicate with each other using requests and responses
Javascript Object Notation (JSON) JSON is a lightweight format for storing and transporting data. It is used for sending data from a server to a client in most modern web applications.
eXtensible Markup Language (XML) XML is a markup language much, like HTML which is used web pages. Like JSON, it was designed to store and transport data.
Open Authorization (OAuth) OAuth is a standard for access delegation, commonly used as a way for internet users to grant websites or applications access to their information on other websites but without giving them the passwords. This mechanism is used by companies such as Google, Facebook, Microsoft, and Twitter to permit the users to share information about their accounts with third-party applications or websites.

FHIR includes a data model in the form of resources and a specification for the exchange of these resources via a RESTful API as well as messaging and documents.

  • Resources span many concepts including the individual (e.g., Patient), workflow (e.g., Appointment), patient management (e.g., Encounter), clinical treatment (e.g., Condition, Observation, MediationRequest, CarePlan, Communication), and financial transactions (e.g., Coverage, Claim).
  • Messages are used to communicate multiple resources in a single exchange.
  • Documents focus on persistence when data touches multiple resources.

It is also notable that the content in the core specification is meant to cover the top 80% of use cases. Unlike older standards (i.e., HL7 V2), the extensions are cleaner and easier. The idea is to focus on the real needs of the industry but allow for all the special use cases as well. More recent standards (i.e., HL7 V3) were commonly criticized for trying to account for too many use cases in the core standard, thus leading to an overly complex standard that was difficult to achieve.

Government Mandates

In 2020, the U.S. Centers for Medicare & Medicaid Services (CMS) finalized a mandate for the use of FHIR by a variety of CMS-regulated payers and providers beginning in mid-2021. These rules have far reaching implications for healthcare providers, payers, and health IT vendors.

  • Health IT Vendors must have a FHIR API that allows patients access to their data.
  • Providers must have FHIR-based APIs to share data with payers and the patient's other providers.
  • Payers must have FHIR-based APIs that serve as a provider directory as well as payer-to-payer exchange.

There are some exceptions to these rules. For further information, see the CMS Interoperability and Patient Access final rule and the Office of the National Coordinator [ONC] Cures Act Final Rule.

Clinical Quality Language (CQL)

Clinical Quality Language (CQL) is a language for representing clinically-focused logic. It is currently being used in a broad variety of applications from quality measurement, to clinical decision support, and public health reporting.

The specification includes a machine-readable representation called Expression Logical Model (ELM) which should enable sharing between heallthcare organizations.
